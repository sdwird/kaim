Engine Architecture
Built for Performance
kaim isn't just an aimbot. It's a meticulously optimized Lua engine built to bypass common execution limits while maintaining flawless frame rates.
01

Zero-Bridge Drawing Proxy

Most scripts update the C++ Drawing API every single frame, causing severe memory bridging latency and inevitable micro-stutters that compound during intense fights.

The custom ND() proxy caches all properties locally in a Lua table. On each __newindex call, it compares the new value against the cached one. Only when they differ does it dispatch the update across the Lua→C++ bridge. This eliminates 90%+ of redundant bridge calls, keeping frame times rock-solid even with 20+ ESP objects on screen.
02

Raycast Allocation Caching

"Smart Mode" checks up to 6 different body parts per player per frame. Standard aimbots instantiate a new filter array for each check, flooding the garbage collector.

Kaim caches the FilterDescendantsInstances array via a _lastVisChar reference. If the current character matches the last one checked, the script reuses the identical C++ allocation across all 6 body part raycasts. This cuts allocation costs by 6x per player per frame and drastically reduces GC pressure.
03

Global Descendant Pre-Pass

To prevent crashes when a player dies, the script checks if their character still exists in the workspace. Standard scripts do this check separately in ESP, Aiming, and HUD code.

Kaim runs IsDescendantOf(workspace) exactly once per player per frame during the pre-computation loop. The boolean result is cached to cd._wsValid and consumed by all downstream systems. This halves the overhead of workspace verification across the entire pipeline.
04

Heartbeat Hitbox Bypass

Nested loops checking thousands of parts on Heartbeat will instantly tank your FPS. Even when the hitbox expander is off, most scripts still iterate every frame.

Kaim implements a strict _hbActive state flag. When the Hitbox Expander is disabled and all parts have been restored, the script permanently halts all hitbox math on Heartbeat. It executes zero code per physics tick until you explicitly re-enable the feature — saving precious CPU budget for other systems.
05

Dynamic Target HUD Engine

4 unique layouts (Premium, Competitive, Compact, Minimalist) that feel completely different but share one unified rendering engine underneath.

Instead of creating/destroying drawing objects when swapping styles, the HUD recycles the exact same 15 drawing objects. Unused elements are hidden through the proxy's visibility flag rather than deleted, guaranteeing zero GC stutter during live style switches. The proxy's __index fallback safely queries C++ properties like TextBounds for pixel-perfect right-alignment without expensive string math.
-- Zero-Bridge Proxy Core
local function ND(t)
  local obj = Drawing.new(t)
  local cache = {
    Visible = false,
    Color = Color3.new(),
    Size = 12,
  }
  return setmetatable({}, {
    __newindex = function(_, k, v)
      if cache[k] ~= v then
        cache[k] = v
        obj[k] = v -- bridge call
      end
    end,
  })
end
